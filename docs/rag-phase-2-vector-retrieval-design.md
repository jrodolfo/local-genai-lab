# RAG Phase 2 Vector Retrieval Design

Status: Implemented for local in-memory vector retrieval and opt-in Qdrant
vector retrieval.

This note documents the phase-2 direction and current implementation for
vector-database-backed RAG retrieval. It expands
[ADR 0013](./adr/0013-use-ollama-embeddings-and-qdrant-for-phase-2-rag-vector-retrieval.md)
with implementation details.

The current implementation is intentionally local and lightweight:

- fixed local corpus rooted at `docs/`
- chunking over project documentation and ADRs
- in-memory lexical retrieval through `InMemoryLexicalRagRetrievalStore`
- optional in-memory vector retrieval through Ollama embeddings
- optional Qdrant-backed vector retrieval through `RAG_VECTOR_STORE=qdrant`
- answers generated by the selected model provider with cited source chunks
- separate `/api/rag/*` endpoints and separate RAG session mode

The RAG UI can choose `Lexical`, `Vector - In Memory`, or `Vector - Qdrant` per
question. `RAG_RETRIEVAL_MODE` and `RAG_VECTOR_STORE` still define the backend
default target when a request does not choose one explicitly.

## Goal

Support vector retrieval with an external local vector database without
replacing the current lexical baseline.

The desired lab shape is comparison:

- `lexical`: dependency-free baseline, easy to inspect and explain
- `vector`: semantic retrieval for questions that use different wording than
  the source documents

The current UI can show which retrieval mode is active and can compare lexical,
in-memory vector, and Qdrant vector retrieval targets for the same question.

## Key Concept

Vector retrieval needs embeddings.

An embedding is a numeric representation of text meaning. The system needs one
embedding for every indexed document chunk and one embedding for the user's
question. Similarity search compares those vectors to find the chunks whose
meaning is closest to the question.

The embedding runtime and the answer provider are separate choices:

- the embedding runtime converts text into vectors for retrieval
- the answer provider generates the final natural-language response

For example, the system can use Ollama with `nomic-embed-text` for embeddings
and still use Ollama `llama3:8b`, Bedrock, or Hugging Face to generate the
final answer.

For operational checks and common local fixes, see
[rag-troubleshooting.md](./rag-troubleshooting.md).

For repeatable lexical vs vector observations, use
[rag-retrieval-evaluation-template.md](./rag-retrieval-evaluation-template.md).

For the concrete Qdrant execution checklist, see

## Embedding Runtime

Current default:

- runtime: Ollama
- model: `nomic-embed-text`

Reasons:

- local-first
- no cloud credentials required
- small enough for a local developer machine
- already aligned with the default Ollama-based local workflow
- already available in the current Mac Mini M4 Pro development environment

Alternatives still worth evaluating later:

- Bedrock embeddings for a managed AWS-aligned option
- Hugging Face embeddings if the remote provider API path becomes important for
  retrieval experiments

Ollama embeddings remain the default because they keep phase 2 local and avoid
mixing vector retrieval behavior with cloud credential setup.

## In-Memory Vector Mode

The in-memory vector mode is useful as a lightweight comparison target:

- it uses Ollama embeddings
- it stores vectors in backend memory
- it supports local lexical vs vector comparison without Docker or Qdrant
- it proves embedding and retrieval boundaries without requiring vector
  database setup

Current limitation:

- the vector index is not durable
- there is no external collection to inspect
- there is no external vector database health check
- large corpora would not be appropriate for backend memory

This mode remains available as a lab baseline after Qdrant. It is useful for
quick local experiments and for debugging whether a failure is caused by Qdrant
setup or by retrieval behavior itself.

## Implemented Vector Database

First external vector database:

- Qdrant

Reasons:

- practical Docker-based local setup
- common fit for small RAG experiments
- simpler first step than operating OpenSearch or another heavier managed
  search stack
- keeps vector storage outside the Spring Boot process while still staying
  local-first

Qdrant is implemented as a second vector store option, not as a replacement for
the current in-memory lexical or in-memory vector stores.

Retrieval store options:

- `lexical-in-memory`: current dependency-free baseline
- `vector-in-memory`: current Ollama embedding experiment
- `qdrant`: durable local vector database backed by Qdrant

The user-facing retrieval target can be `lexical`, `vector:in-memory`, or
`vector:qdrant`. The backend startup settings remain split between retrieval
mode and vector store so vector mode can use memory or Qdrant by default.

## Index-Time Flow

At index time:

```text
docs text
  -> chunking
  -> Ollama embedding model, such as nomic-embed-text
  -> vectors
  -> Qdrant collection
```

Each stored vector keeps metadata needed for citations and debugging:

- source path
- chunk id
- chunk text or excerpt
- document title when available
- embedding model name
- corpus root
- index timestamp or version
- content hash when available

The embedding model name matters because changing the embedding model requires
reindexing. Vectors generated by different embedding models must not be mixed
in the same collection unless the system explicitly supports that.

## Qdrant Collection Shape

Default collection:

```text
local_genai_lab_docs
```

Point id:

```text
stable hash of source path + chunk id + embedding model
```

Payload fields:

- `source_path`
- `chunk_id`
- `title`
- `text`
- `corpus_root`
- `embedding_provider`
- `embedding_model`
- `indexed_at`
- `content_hash`

The collection is treated as disposable in the current implementation.
`Rebuild Index` deletes and recreates the collection instead of trying to do
incremental updates. Incremental indexing remains a future option if the corpus
grows or rebuild time becomes a real problem.

## Query-Time Flow

At query time:

```text
user question
  -> same Ollama embedding model used at index time
  -> query vector
  -> Qdrant similarity search
  -> top matching chunks
  -> selected answer provider
  -> answer with cited source chunks
```

The query must use the same embedding model family as the indexed chunks. The
document vectors and query vector need to live in the same vector space for
similarity search to be meaningful.

The selected answer provider remains independent. Retrieval can use Ollama
embeddings while answer generation uses the provider/model selected in the RAG
workspace.

The Qdrant-backed retrieval store is available behind
`RAG_RETRIEVAL_MODE=vector RAG_VECTOR_STORE=qdrant`. When `Rebuild Index` runs
in that mode, the backend embeds the fixed docs corpus, recreates the configured
Qdrant collection, and upserts the embedded chunks with citation payloads.
Qdrant mode does not silently fall back to lexical or in-memory vector
retrieval.

## Configuration Shape

These settings define the current vector retrieval surface.
`RAG_VECTOR_STORE`, `RAG_QDRANT_URL`, and `RAG_QDRANT_COLLECTION` are visible in
configuration/status when Qdrant mode is selected.

```properties
RAG_RETRIEVAL_MODE=lexical
RAG_AVAILABLE_RETRIEVAL_MODES=lexical,vector
RAG_EMBEDDING_PROVIDER=ollama
RAG_EMBEDDING_MODEL=nomic-embed-text
RAG_VECTOR_STORE=in-memory
RAG_QDRANT_URL=http://localhost:6333
RAG_QDRANT_COLLECTION=local_genai_lab_docs
RAG_VECTOR_TOP_K=4
```

The existing `RAG_TOP_K` can remain the shared retrieval limit if there is no
reason to split lexical and vector limits.

Explicit Qdrant backend run:

```bash
RAG_RETRIEVAL_MODE=vector RAG_VECTOR_STORE=qdrant ./scripts/restart.sh
```

`RAG_VECTOR_STORE=in-memory` remains the low-friction vector-store default for
local comparison. `RAG_VECTOR_STORE=qdrant` selects the durable vector-store
path and requires Qdrant to be reachable before rebuilding or querying the
Qdrant index. The UI can also target `vector:qdrant` per question or through
`Compare Retrieval Targets`.

## Local Qdrant Runtime Shape

The first local Qdrant setup is Docker-based.

Explicit Qdrant backend flow:

```bash
RAG_RETRIEVAL_MODE=vector RAG_VECTOR_STORE=qdrant ./scripts/restart.sh
```

The normal `./scripts/start.sh` and `./scripts/restart.sh` scripts also try to start the local
Qdrant Compose service by default when RAG is enabled, because the UI exposes
`Vector - Qdrant` as a comparison target. If Docker is unavailable during normal
lexical startup, the app can still run and the Qdrant target remains
unavailable until Qdrant is started.

The application status makes Qdrant reachability visible for the optional
`Vector - Qdrant` comparison target and for explicit Qdrant backend mode.
Example `./scripts/status.sh` lines:

```text
backend rag retrieval mode: vector
backend rag vector store: qdrant
backend rag qdrant url: http://localhost:6333
qdrant service: ok
qdrant collection: present (points=123)
```

If Qdrant is unavailable, the UI shows a clear operational message instead of
silently falling back to lexical retrieval. Silent fallback would make
evaluation misleading.

For direct browser and `curl` inspection of the local Qdrant collection, see
[rag-qdrant-inspection.md](./rag-qdrant-inspection.md).

## API And UI Expectations

`/api/rag/status` exposes enough state for troubleshooting:

- RAG enabled state
- active retrieval mode
- vector store
- embedding provider
- embedding model
- index status
- document count
- chunk count
- Qdrant URL when Qdrant is configured
- Qdrant collection status for Qdrant readiness and comparison

The RAG page keeps the main interaction simple. Detailed status stays compact,
but vector database failures are actionable through the status strip, query-card
readiness hint, and comparison result errors.

Good UI messages:

- `Qdrant is not reachable at http://localhost:6333. Start it and rebuild the index.`
- `The Qdrant collection is missing. Click Rebuild Index.`
- `The index was built with a different embedding model. Rebuild the index.`

Avoid vague messages such as:

- `RAG failed`
- `No results`
- `Backend error`

## Implemented Test Coverage

Backend unit tests:

- collection name and payload mapping
- Qdrant client request construction
- query vector search maps Qdrant points back to `RagMatch`
- missing collection produces a clear status
- indexing and retrieval service boundaries

Backend integration posture:

- keep default tests independent from Docker/Qdrant
- keep Qdrant request construction and failure behavior covered without a live
  container by default
- add live Qdrant integration tests later behind an explicit profile or
  environment flag if needed

Frontend tests:

- render Qdrant status and readiness hints
- show actionable Qdrant unavailable message
- keep lexical mode UI unchanged
- keep vector in-memory mode UI unchanged
- render comparison results with partial Qdrant failure

Script tests:

- `./scripts/start.sh` tries to start Qdrant for the optional comparison target
- `./scripts/start.sh` fails strictly when explicit Qdrant backend mode cannot start
- `./scripts/status.sh` reports Qdrant readiness for optional comparison and explicit
  Qdrant backend mode
- `./scripts/status.sh` does not require Docker/Qdrant for lexical mode
- `./scripts/restart.sh` preserves explicit `RAG_VECTOR_STORE=qdrant`

## Implemented Migration Path

Completed sequence:

1. keep `InMemoryLexicalRagRetrievalStore` unchanged
2. keep the current in-memory vector store available
3. add a Qdrant client boundary
4. add a Qdrant-backed vector store implementation
5. add Docker Compose support for local Qdrant
6. add a vector indexing service for the existing `docs/` corpus
7. expose vector store, Qdrant URL, collection, and readiness through
   `/api/rag/status`
8. update `./scripts/status.sh` to report Qdrant readiness for optional comparison and
   explicit Qdrant backend mode
9. add backend tests for Qdrant payload mapping, indexing, and retrieval
10. add frontend tests for Qdrant status and failure messages
11. document the manual lexical/vector/Qdrant comparison flow

## Why Lexical Remains Available

Lexical retrieval should remain available even after vector retrieval exists.

Reasons:

- it is dependency-free and works when Ollama embeddings or the vector database
  are unavailable
- it provides a baseline for evaluating whether vector retrieval actually
  improves the project
- it is easier to explain in documentation and interviews
- it helps debug whether a weak answer is caused by retrieval, prompting, or
  the selected answer model
- it keeps the lab honest by making retrieval tradeoffs visible instead of
  assuming vector search is always better

## Boundaries To Protect

Keep these outside the current vector retrieval slice:

- document uploads
- automatic routing between normal chat, tools, and RAG
- MCP tool retrieval
- report artifact retrieval
- production multi-user indexing
- replacing lexical retrieval
- removing the in-memory vector store

Those are useful later directions, but they would make the current RAG boundary
too large and harder to evaluate.

## Risks

- embedding model changes require reindexing
- vector database setup adds an operational dependency
- vector search can retrieve semantically plausible but source-weak chunks
- citation quality must still be validated against the actual retrieved chunks
- answer generation can still hallucinate if the prompt does not enforce source
  grounding clearly
- silent fallback from Qdrant to lexical would make comparisons invalid
- persisted vectors can become stale if corpus changes are not reindexed

## Open Questions

- Should scripts support an already-running external Qdrant instance with the
  same first-class behavior as the local Docker Compose service?
- Should a future implementation preserve the collection and upsert changed
  chunks instead of deleting and recreating it on every rebuild?
- Should `/api/rag/status` expose the embedding model used for the current index
  separately from the configured embedding model?
- Should the UI compare embedding models as well as retrieval stores?
- What repeated evaluation failures would justify changing chunking, scoring,
  or the default retrieval target?
